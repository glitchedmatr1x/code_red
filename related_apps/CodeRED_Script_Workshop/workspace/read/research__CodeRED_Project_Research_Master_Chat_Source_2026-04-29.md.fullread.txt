## 0. Executive Summary

The strongest current direction is no longer “replace a prop with a car model.” The Blackwater WSI test proved the WSI reference can be changed, because the original broken car disappeared. The replacement did not appear because `car01` is not a valid standalone static prop in that placement slot. That points toward the **Gringo vehicle system** and **script/mission vehicle generator path**.

The most important research lanes are now:

1. **WSI semantic editing**
   - WSI controls world sectors, bounds, props/drawables, children, locators, and map-layer streaming.
   - CodeX confirms WSI is `Rsc6SectorInfo` / `sagSectorInfo`, resource type `134`.
   - Code RED now has tools for WSI scan/export/array walking.

2. **Vehicle spawning through Gringo dictionaries**
   - `commongringos.wgd` contains vehicle gringos such as `Vehicle_Generator`, `car_gringo`, `PlayerCar`, `CarCrank_gringo`, and `Gen_Vehicle_Brain`.
   - FBI/mission tokens like `VEHICLE_Coach01`, `VEHICLE_Cart01`, `VEHICLE_Wagon02`, and `VEHICLE_WagonPrison01` appear in the vehicle/gringo research path.
   - The next best test is **WSI ↔ WGD Gringo correlation**, not another raw model-name swap.

3. **Map expansion lane**
   - WSI connects to WVD/WBD territory assets, WTB terrain boundaries, WTL terrain LOD world resources, mapres textures, and map states.
   - CodeX can view/edit pieces of these systems and should guide Code RED’s Python structure ports.
   - The practical expansion path is to understand/patch sector/terrain relationships conservatively before touching geometry.

4. **Cutscene/FBI mission vehicle research**
   - Cutscenes include staged vehicle-related mission context, especially FBI missions.
   - Cutbin data is useful for actor/prop/camera/timeline identity but is not yet a direct persistent-world spawn source.
   - The useful crossover is correlating FBI cutscene/mission names with WSC/WGD vehicle setup paths.

5. **Multiplayer lane**
   - Current content already has the front-end bridge to `StartMultiplayer`, `RequestJoin`, `HudSceneOnline`, and LAN/public/private menu flows.
   - Mixed-version donor files show auth-bypass and LAN-flow ideas but should not be wholesale merged.

## 1. Current Working Principles

### 1.1 Do not fake unresolved systems

Code RED passes should preserve the distinction between:

- **Confirmed editable:** known XML/tune/plain text/resource references.
- **Confirmed decodable:** WSI/WGD/RSC payloads where structure is partially known.
- **Replaceable but not semantic:** WVD/WBD/WTD/WCDT resources where full internal edit support is not done.
- **Research-only:** WSC/SCO compiled scripts, cutscene timelines, multiplayer runtime state.

### 1.2 Patch copied archives only

For risky resource experiments:

- Never patch originals in place.
- Always build a copied output archive.
- Keep proof JSON.
- Reopen the patched archive and verify decoded target bytes.
- Use one change at a time.

### 1.3 WSI first, then WGD/Gringo, then deeper script integration

The safest current ladder is:

1. WSI scan/export.
2. WSI sector/child/drawable/prop/locator export.
3. WSI ↔ WGD gringo reference correlation.
4. Single test placement changed to a known gringo path.
5. Mission/FBI script/cutscene vehicle correlation.
6. Only then attempt vehicle generator behavior patches.

### 3.3 CodeX Gringo / WGD Support

Source:

```text
https://github.com/Foxxyyy/CodeX.Games.RDR1
```

Key CodeX file:

```text
Files/WgdFile.cs
RSC6/Rsc6Gringo.cs
```

Important facts:

- `.wgd` is `WgdFile`.
- `WgdFile` wraps `Rsc6GringoDictionary`.
- `Rsc6GringoDictionary` contains `Hashes` and `Gringos`.
- Gringo base components expose query names, hash codes, parent components.
- Item gringos expose script names, gringo names, child components, instanced items, activation radius, slot counts, critical/large/maintain-state flags.
- Use contexts expose local position, facing, radius, use flags, physics/nav flags, player usability, and movement behavior.

This confirms Gringo dictionaries are likely the correct layer for vehicle behavior and interaction.

### 6.2 WSI structure status

WSI is not just a hash list.

It includes:

- Sector hierarchy.
- Names/scopes.
- Hashes.
- Bounds.
- Child sectors.
- Drawable instance references.
- Prop references.
- Locator references.
- Visual and bound dictionary references.
- Flags and resident/disabled state.
- RAGE virtual pointers.

Key internal pointer pattern:

```text
0x50000000 + payload_offset
```

Safe edit direction:

- Start with coordinate-only fields only after verified.
- One changed placement at a time.
- Copy-only RPF patches.
- Keep WVD/WBD as view/extract/replace-only until fully decoded.

### 11.2 Where is the vehicle token parameter?

Candidates:

- WGD instanced item attributes.
- WGD ItemAttributes child.
- WSC mission script.
- Cutscene RBF/cutbin event data.
- Gringo brain script resource.
- Tune/refgroup/event setup.

### 11.3 Are cars treated as vehicles, props, fragments, or scripted mission actors?

Blackwater test suggests:

- Not a simple WSI static prop.
- Probably a gringo/script vehicle object requiring behavior setup.

### 11.4 Can CodeX export WGD as editable text?

The CodeX `WgdFile` has `Read`/`Write` meta behavior. It may be able to display/edit some gringo data as text, but Code RED should first export safely and compare structure before writing WGD.
